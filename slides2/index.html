Konflikt

Keď dvaja zmenia to isté miesto

Rôzne súbory git spojí sám (to už vieš z minula). Ale ten istý riadok v tom istom súbore? Git nevie rozhodnúť, ktorá verzia je správna — spýta sa teba.
Konflikt nie je chyba. Je to rozhodnutie pre človeka.
Ako konflikt vyzerá

Vyrob a vyrieš konflikt

Vo dvojiciach zmeňte ten istý riadok. Jedna pushne, druhá pri pulle dostane konflikt → otvor súbor → vyber verziu → ulož → commit.
→ GitKraken · repo  wta-a-24 · vo dvojiciach
Branch (vetva)

Aby spoločná verzia ostala v bezpečí

Vlastná bezpečná línia, kde robím bez zásahu do spoločnej verzie.
Dve vetvy, dva účely
main
stabilná spoločná verzia
nepracujem do nej priamo
feature vetva
moja pracovná línia
nové commity idú sem
Názov podľa obsahu: add-about-page, fix-header-typo
Životný cyklus vetvy

Odbočím, commitnem, zlúčim späť

main beží ďalej · vetva si žije svoj život · na konci sa vráti do main

Vytvor vetvu, commitni, push

→ GitKraken: pull → Branch add-tvojemeno → commit → push
Merge

Vetva sa spojí späť do main

⚠️ Najprv checkout na cieľovú vetvu (napr. main) — až potom do nej zlúč svoju vetvu.
Ak sa dotklo to isté miesto, vznikne konflikt — a ten už vieš vyriešiť.
Hands-on · 3

Zlúč vetvu do main

V GitKrakene: checkout main → pravý klik na vetvu → Merge into main. V grafe uvidíš, ako sa línie spoja.
→ GitKraken: checkout main → Merge into main

Merge cez GitHub, s review

add-stano → main

Načo je PR

Tri dôvody pre pull request

Reviewniekto sa pozrie skôr, než to ide do main
Spätná väzba a kvalitapriestor navrhnúť úpravy
História „prečo"zostane zapísané, čo a načo sa menilo
Hands-on · 4

Otvor PR, daj review, zlúč

Push vetvy → GitHub Compare & pull request → názov + popis → vo dvojiciach review → Merge → v GitKrakene pull na main.
→ GitHub: PR + review + Merge · GitKraken: pull
